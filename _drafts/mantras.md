Keep these ideas in mind in your day-to-day workflow.


## Software Development Mantras

### Testing

> Move fast and break things. - Some random developer

We manage the complexity of a codebase by writing tests. Without backing
tests, a feature will ultimately fall victim to rot and decay.

Software is in constant state of flux. We add new features to our codebase on a
hourly basis. **We can easily break existing features when adding new ones. This
is unintentional, but it happens**. Without tests, there is no notification that
an existing feature has been compromised. **We can easily and unknowingly
destroy someone else's work when tests are not in place.**

If you want your feature to last more than a few hours, there _must_ be tests to
describe it.

* Write both unit and feature tests.
* Write feature tests for the "happy path."
* Write feature tests for the "sad path."
* Write unit tests for edge cases.
* When fixing a bug, write a feature test to describe the desired functionality.
Then, fix the bug.
* Fill in existing features with tests when refactoring or adding new functionality.
* If a new feature will break existing functionality that does not have tests in
place, let it. If the feature is worth keeping, someone will let us know that it
is broken.
* Tests are a way to offload cognitive load. Let's use testing as a way to
alleviate the worry of breaking the application.

### Code Review

> Debugging is twice as hard as writing the code in the first place. Therefore,
> if you write the code as cleverly as possible, you are, by definition, not
> smart enough to debug it. - Brian Kernighan

In order to minimize our [Bus Factor](https://en.wikipedia.org/wiki/Bus_factor),
and to ensure that features in our codebase can be understood by more than one
developer, we practice peer review for all pull requests.

* All pull requests must be reviewed and understood by another developer.
* When the intention of a pull request is not clear, ask the author to pair on
reviewing the code.
* Use your downtime to improve your skillset.
  - Read an article.
  - Read code documentation.
  - Write an article for your blog.
  - Write an article for your company's blog.
  - Work through coding challenges.

There are plenty of asynchronous tasks that you can, and should, perform to
improve your abilities while waiting for your code to be reviewed.

### Problem Solving

As developers, we are problem solvers. It can be daunting when we are presented
with an endless series problems to solve, and are rarely congratulated for our
efforts.

In order to keep our morale up, and be the best problem solvers we can be,
follow these guiding principles...

* Break problems into smaller, more manageable pieces.
* Ask for help when a problem seems too complex.
* Ask clarifying questions when a problem is fuzzy.
* Follow "[The Path](https://gist.github.com/radavis/9bafb2930fa3f6f9e441782cf7631250)": User Story -> Acceptance Criteria -> Feature Test
* Congratulate your peers for their work and effort.


## Workplace Goals

### Daily Goals

* Get AFK (away from keyboard), and go for a walk at least once a day for 15
to 20 minutes, during working hours.
* Acknowledge hard work and effort by thanking your coworkers for a job well done.

### Weekly Goals

* Pair with someone for an hour. You can pair on problem solving, algorithm
development, writing code, software architecture, code review, or learning.
* Spend 20% of your working hours on personal development. Learn something new
or do a deep-dive into a technology you want to learn more about.
* Wednesdays are for heads-down work. Meetings should not be scheduled for
Wednesdays, except for extenuating circumstances.

### Monthly Goals

* Present on a topic in a Skill Share.
* Update your calendar to take a Monday and/or Friday off for a long weekend.

### Yearly Goals

* Attend a conference in another city.
* Plan (and go on) a vacation that lasts **at least a week**.
* Utilize your health care benefits.
  - Schedule a yearly physical with your primary care physician.
  - Schedule a yearly eye exam.
  - Schedule routine cleanings with your dentist, twice a year.
  - Visit a psychiatrist or psychologist on a regular basis.
  - Put these appointments on your shared [Google Calendar](https://calendar.google.com/calendar/render) and label them as "Doctor Appt." or "Busy", so we can plan around them.
