## Synopsis

The world of goverment contracting demands results, often at the expense of code quality. Component names become meaningless after multiple refactors. Tests become illegible, 500-line file behemoths, which take hours to understand. REST-ful resources become REST-less endpoints whose names no longer equate to their intended purpose. Slicing the software stack is an exercise for experts in memory recall, since the names of files, functions, classes, tables, and components have little correlation, and only some organization. As a contractor, you're often dropped into the middle of a project such as this. You ramp up on the tech. You crank out features. There is time for some refactoring; typicaly only enough to make a dent. Then, after 14 two-week sprints, you're dropped into another project of similar quality.

After living this experience for two years, I'm ready for something different. Your codebase doesn't have to be perfect, but my intentions will be to pull it into a direction that is less confusing, more organized, and more beginner-friendly. If you require a fresh software solution, my focus will be on creating reasonable, testable, and repeatable patterns within the codebase. When it comes time to augment the codebase for new functionality, there will be a clear path forward with patterns for others developers to follow.

I love writing tests. Unit tests are great for catching edge cases. I find comfort in writing tests that exercise the entire stack, and ensure a feature works as intended. When I ramp up on a new project, the first place I go is to these "executable specifications" to understand the intentions of the codebase.

Helping others level up is another love of mine. Code reviews and mentorship are effective ways to achieve this goal. I enjoy learning and sharing what I learn. A primary goal of mine is to ensure the code I write is easily understood. Naming is important. Tests are important. File organization is important.

I'm most comfortable in the Rails framework, because of its thoughtful conventions. I'm also well-veresed in React, which I have used across a large number of projects. I'm open to learning new technologies, and writing articles to make the learning path a little less rough for others, provided time and space are given to achieve that goal.

If you're looking to augment your software team with someone who wants to lift up the skills and abilities of others, look no further. If you want to create a new software platform that other developers will enjoy working on, I'm your guy.

Tell me about your organization's software goals, and let's chat about how I can help you to achieve meaningful progress towards those goals.


## Email Campaign - Do what you do: describe software so anyone can understand.

When building a software solution, It is necessary to discover and document reproducable patterns for slicing the software stack. A web application consists of many layers of technology, stitched together to serve a business' needs. Typically, technologies used within a Web Application are grouped into one of two following categories: front-end UI tech, or back-end API tech.

When a feature is requested, developers must survey the available tech, plan the work to be done across the software stack, and then execute. If your codebase is in good health, with identifiable patterns, this process is usually quick and painless. Often it is possible to "follow by example." Past pull requests, or merge requests, will outline the changes necessary to each layer. Tickets are completed in a timely manner, and features are pushed out to production on a regular basis.

The signs of a codebase in poor health are easy for anyone involved to identify, regardless of an individual's technical know-how. Developers will seem frustrated with their ability to deliver features in a timely manner. Seemingly simple asks take multiple sprints to complete. Unexpected problems crop up, pushing items on the roadmap further into the future.

If you see these warning signs, your application's health is waning. Perhaps it's already on its death bed, waiting for someone to pull the plug.
